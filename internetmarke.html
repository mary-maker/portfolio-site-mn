Redesign of “Internetmarke” configuration

Deutsche Post

About

The “Internetmarke” function on Deutsche Post Shop enables the user configurate and buy self-printable postage stamps. The previous user flow is considered in lacking in several aspects like intransparency of prizes or wrongly used interaction patterns.
We redesigned the configurator and optimized the user flow starting with a Google design sprint and following up with additional iterations loops.

Time

The design process took place within 10 weeks. The launch of the redesigned “Internetmarke” configurator was on March, 21th 2019.

Four out of five team members are playing “Rock, Paper, Scissors” during a retrospective meeting.

my tasks

Design sprint facilitator (organizing location and equipment, as well as moderating the sprint), UX Designer.

tools

A ton of Post-Its, Sketch, Axure RP, InVision, Zeplin.

Prototype of the first concept.

1st iteration

The Design Sprint helped to create a prototype and an usability test within five + one days. We added an additional day to the usual five-day-sprint, giving extra time for the prototyping task and QA.

The prototype in action: the first usability test.
Protocolling the user feedback via color-coded Post-Its.
After clustering the user feedback we reduced the reoccuring usability problems to specific UI areas.

2nd iteration

After the successful Design Sprint we tackled the occuring usability problems of the first concept. Also we added functions like “choosing the paper” which couldn’t be tested in the previous user test. 
The following prototype was colored in brand yellow instead of the wireframing blue/grey. In doing so we checked whether the color scheme is distracting the user from call-to-actions and whether the user is still able to spot the functions and understand their behavior.

3rd iteration

We were on a good track with our concept. The core of the configuration flow was the following: every added postage stamp to the cart will be automatically placed on one PDF sheet. So the user won’t deal with any other configuration except for the stamp itself.

Well, it happened we were missing important information about the backend. We were confronted with a massive backend modification for our “one-PDF-automatically”-plan.
After several talks with programmers, managers and so on, we knew tweaking the backend was not an option. Our concept wasn’t going to work with the actual backend.
 So we dumped it and did a restart.

Faster results

We focused on getting user feedback as quick as possible and decided to go with a paper prototype.

Previous learnings

The feedback of the two previous usability tests came in handy for designing functions and for choosing suitable interaction patterns.

Button vs. button

The usability tests 3 and 4 were essential for tweaking the two most important CTAs (call-to-actions).

Learnings of previous usability tests 1 and 2 placed on the design.
The 4th prototype uses mainly patterns and colors of the current CSS styling. Some components got a redesign e.g. the tooltip icon (regularly yellow colored on DP’s websites).

Ready for development

After the four usability tests we were ready to put the concept into the real thing.

Zeplin was the favored platform to deliver styling and dimensioning to the developers.
